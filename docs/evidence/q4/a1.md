# TL;DR

- Self-hosting costs ~$58k over 3 years for 200 users. ChatGPT Enterprise at $60/seat/month costs $432k over the same period. The economics alone close the debate.
- Five of our seven MCP tools target RFC 1918 addresses or IAM instance-scoped credentials. No public cloud AI product can reach those endpoints i.e. not a missing feature, but a routing invariant. The `ssh-exec` and `aws-resources-operations` tools are the core of the compliance audit workflow; removing them means we are not comparing equivalent products.
- Our RAG pipeline (pgvector embeddings, `user_memories*` semantic graph, knowledge bases) has no migration path to ChatGPT Enterprise. LobeChat's lock-in is technical and reversible with a `pg_dump`; ChatGPT's is contractual and architectural.

# 3-year TCO comparison (200 users)

Using the interpolated Q3 costs for 200 MAU: infrastructure cost ≈ $490/month + Bedrock tokens at 2x the cost of 100 MAU ($192/month) = **$682/month**. Operations overhead comes from deployment logs and the most time-consuming part seems to be `patches/route.js` compatibility. Each update of the LobeChat images breaks Next.js webpack chunk IDs and silently causes MCP call failures: the tRPC route will return HTTP 200 but MCPHub won't receive anything and the returned model results will contain hallucinations for the called tools. Each image upgrade requires 3–4 hours to re-extract and re-apply the patch, at roughly one release per month, this alone accounts for the bulk of ongoing ops time. When added to EC2 updates, Qdrant snapshots verification, and quarterly secrets rotation, this gives us an estimate of around 10 hours per month of operations.

# Lock-in analysis

**Data.** The most difficult lock-in is the embedding space. We have `embeddings` table with 1024-dimensional float arrays tied via six-level RAG pipeline (`knowledge_bases → knowledge_base_files → files → file_chunks → chunks → embeddings`). Those are model-specific vectors – switching to ChatGPT Enterprise requires us to discard everything we've accumulated so far and re-ingest the sources from scratch, using OpenAI vector dimensionality. Moreover, `user_memories*` is a five-level semantic graph structure (memories, contexts, experiences, identities, preferences) that uses HNSW index and access scoring. ChatGPT Memory is a text list, without even a bulk-export API endpoint. On our end, we just `pg_dump` all of it – instantly, at any time.

**Identity.** Casdoor is OIDC-compliant and supports federated logins from practically any IdP (e.g. Cognito, Okta, Azure AD) within hours. ChatGPT Enterprise is locked into OpenAI's organisation, while Copilot is bound to Microsoft tenant – neither one allows for programmatic user export.

**Model.** Our stack allows inference through either OpenRouter or Bedrock, which means access to Claude, GPT-4, Llama and Mistral with literally one env-var change. ChatGPT Enterprise locks you down to OpenAI's release cycle – once GPT-4o gets deprecated, you're stuck with whatever follows, paying whatever price they ask for.

**Tooling.** Five out of seven MCP tools are incompatible with SaaS by design – covered in Capability Gap section.

# Data sovereignty + compliance

Our S3 bucket in `eu-west-1` stores personal data under GDPR: specifically IP addresses in `auth.log` which constitute personal data according to *Breyer*, C-582/14, CJEU 2016, and usernames as direct identifiers. We have an Art. 28 DPA for Enterprise in place provided by OpenAI, but inference still goes through their US-East infrastructure, which is a cross-border transfer and requires SCCs in addition to all the GDPR overhead for every audit.

For financial services firms under **DORA** (EU 2022/2554, effective January 2025), critical ICT providers must show operational resilience while firms must maintain viable exit plans. Using OpenAI to handle the audit compliance evidence pipeline would create exactly what DORA seeks to prevent: reliance on a single vendor, lack of alternative exit vectors, and no SLA based on supervisory standards.

Bedrock Inference self-hosted on AWS stays within the AWS network as the data will not be leaving our VPC at any point in time. The tamper-proof evidence trail, which consists of Playwright timestamping the PDF -> MinIO storing the PDF with its own hash -> S3 Object Lock in COMPLIANCE mode which cannot edit the data -> Postgres' global_files.hash_id with SHA-256, is entirely verifiable without OpenAI or Microsoft.

# Capability gap

**What self-hosted does that SaaS cannot.** Five out of seven MCPs cannot be provided by any cloud service product due to network architecture constraints. `ssh-exec` connects to `10.2.0.3:22` – RFC 1918 Class A range. A TCP SYN sent from any OpenAI IP to 10.x.x.x is immediately rejected by the internet router. The `aws-resources-operations` relies on credentials fetched from the EC2 metadata service (`169.254.169.254`) – non-exportable feature. Providing AWS keys to circumvent that limitation constitutes CIS 1.13 and 1.14 violations we are assessing now. `filesystem`, `playwright`, and `pickstar-2002-minio-mcp` reach out to Docker internal hostnames which are unreachable outside the compose network. Two remaining products (`notion-mcp` and `aws-documentation`) interact exclusively with public HTTPS endpoints. Apart from the toolkit, there is a need for custom RAG using an organization-specific evaluation harness (`rag_eval_*` tables) and complete model portability using OpenRouter.

**What SaaS does that self-hosted does not.** There is no uptime SLA published as of yet, hence reliability is defined by `m5.xlarge` hardware and availability of the person on-call at the time. Neither ChatGPT Enterprise nor Copilot needs a patch compatibility step after each software update. They both feature mobile applications and provide 24/7 enterprise support. M365 integration depth of Copilot becomes truly valuable in general office scenarios. When talking about teams conducting simple knowledge work without private infrastructure available, the case for self-hosting starts to crumble.

# Recommendation

**Host Yourself if:** (a) the use case demands MCP tool operation on private network infrastructure; (b) the chain of custody for audit evidence is required to be independently verified and not dependant on a third party's inference layer; (c) GDPR cross-border transfer limitation applies or DORA concentration restrictions apply; (d) at least 0.2 FTEs can handle image upgrades and infrastructure management.

**Purchase (start with Copilot) if:** the main purpose of your use case is general conversation AI without private infrastructure access; you are below 50 members on your team, and operational management overhead cannot be internalized. Then it would make sense to get Copilot at $30 per seat.

This particular use case-automated SOC 2 audit compliance with MCP tools on private AWS infrastructure, which cannot function on any other architecture.

# Reversal trigger

The recommendation stands on two legs: verifiable private network access and chain of custody. Either alone renders the recommendation invalid.

**Trigger 1: SaaS receives verifiable private VPC connectivity.** Azure Private Link has been rolled out for select Power Platform services from Microsoft. In the event that this feature expands to allow the use of the Copilot AI tools with verified connectivity to RFC 1918 addresses within the customer's VPC, and without the requirement for a customer-controlled intermediary, the routing-invariant argument falls flat.

**Trigger 2: OpenAI releases tamper-proof audit logs with EU-sovereign inference capabilities.** Should an independent auditor be able to verify cryptographically-signed logs that prove evidence was collected by tool X at time T unmodified and inference never crossed into a non-EU jurisdiction, then all arguments based on chain of custody and GDPR compliance become invalid.

Neither one of these developments will happen soon. AI reasoning using Copilot over private VPC connections from Microsoft will not be feasible until 18 to 24 months in the future, and even longer will be OpenAI EU-sovereign inference with audit logs. For the time being, self-hosted is the only possible design pattern.
