# Q1 — Use case design

## Context

This is a mid-sized B2B SaaS company with about 200 people and its own internal cloud team responsible for running its production AWS infrastructure in eu-west-1. The company goes through annual SOC 2 Type II audits and quarterly light checks to be prepared for customer security questionnaires. The problem isn’t about lacking skills. The company has engineers who exactly know how a CIS compliance audit should work. The issue here is that evidence collection is purely mechanical, which means running certain AWS CLI commands, logging in via SSH to EC2 instances and grabbing the configuration files, correlating each finding with the benchmark text, writing the narrative, making screenshots, saving all findings in Notion. In terms of engineering time, one audit cycle takes between 2 and 4 days before the external auditor gets there – at the daily rate of about €600, it translates into €1,200 - €2,400 per audit cycle in saved engineer time alone, excluding the downstream impact of failing on repeating findings. Failing to comply with repeat findings may mean triggering mandatory remediation procedures and risking reputation due to enterprise clients checking the SOC 2 report before signing a contract. This solution will change that by reducing evidence gathering and narrative writing to several minutes.

## Personas

Persona 1 — The Cloud Security Engineer ("The Preparer")

Responsible for the AWS infrastructure on a day-to-day basis and provides the evidence package prior to each audit. Manually runs queries, SSHs into prod machines to inspect the sshd_config, looks through auth logs with grep, and then spends hours converting JSON output into formatted findings aligned with the SOC 2 requirements with screenshots at specified timestamps. The consistent pain point they have is repeat findings. If the security group had the same risk accepted and now it has expired, finding it means going through the audit document from last quarter by hand and they consistently miss it. With our stack, they submit a job to audit in plain English and receive an evidence package filled with data. The restriction that the SSH access be read- only is appropriate to their practice.

Persona 2 — The External Compliance Auditor ("The Tester")

They come for a 2-4 weeks engagement with a test plan of 80-120 control objectives. The current challenge they face is logistics (take around 40% of their time): trying to gather evidence from the internal team and then standardizing artifacts, each control may have a CLI output, the next a screenshot of the console from a different date, and the next may have written documentation by someone without any artifact at all. What frustrates them further is not being able to check anything by themselves and only relying on what is provided to them by the client's exports. This particular stack allows them to have read access to their MCPHub via bearer token and execute queries against the AWS infrastructure of the client. This way evidence is self-collected, tool-timestamped and stored in a MinIO bucket both sides can access.


## Journey

Control: CIS AWS Foundations Benchmark 5.2 - " Ensure no security groups allow ingress from 0.0.0.0/0 to port 22" ([CIS AWS Foundations Benchmark v3.0][1]). SOC 2 mapping: CC6.6 - logical access over infrastructure and software ([AICPA Trust Services Criteria, 2017][2]). Most often raised red flag in mid-tier AWS audit projects. 

The engineer types a single prompt: check all security groups for unrestricted SSH ingress, SSH into affected instances, screenshot the console, write a finding to the Q3 2026 audit Notion page, and archive all artifacts to MinIO. LobeChat sends the job to MCPHub through the bearer token.

Step 1:  aws-documentation fetches the benchmark control text and NIST mapping (AC-17, SC-7) precisely to ensure that the Notion write cites the official text and not just a rephrasing of it. 
Step 2: RAG against historical findings. Before running the queries on the live AWS account, the stack checks pgvector for historical instances of the same control. Past audit findings were embedded using text-embedding-3-small (via proxy via OPENAI_PROXY_URL). RAG finds a Q4 2025 chunk: the same security group already had a risk acceptance with an expiration date of March 2026, but now it's May 2026. Had this RAG not happened, the engineer would have flagged a new Medium issue. But after it did happen, they flagged a repeat High with an expired risk acceptance, and it went down a completely different remediation route. Throughout the journey, reasoning, synthesis, and all written outputs are handled by a frontier model via OpenRouter (defaulting to Claude 3.5 Sonnet or GPT-4o). The only exception is Step 2, where text-embedding-3-small handles the vector query.
Step 3: The command used to query AWS is aws-resources-operations, which uses the boto3 library to execute describe_security_groups on the live AWS account, using credentials provided by ~/.aws. One matching result: sg-0a1b2c3d (prod-bastion-sg). A follow-up describe_instances call identifies the instance: i-0abc123456def, prod-bastion-01, running.
Step 4: ssh-exec connects to 10.2.0.3:22 from inside the Docker network. Four whitelisted commands run: ps aux | grep sshd confirms the service is active; cat /etc/ssh/sshd_config reveals PermitRootLogin yes and PasswordAuthentication yes (two linked CIS violations, 5.3 and 5.4); tail /var/log/auth.log shows active brute-force attempts from external Ips which results in an escalation of severity from Medium to High.
Step 5: playwright screenshots security group inbound rules, instance details from AWS console and generates a combined PDF report at /tmp/CIS-5.2-evidence-2026-05-29.pdf. Artifacts have timestamps from the automation tool and not from the emails sent to the customer.
Step 6: minio-mcp uploads all artifacts to audit-evidence/2026/Q3/CIS-5.2/. MinIO metadata (time created, ETag) gives proof against any tampering. 
Step 7: notion-mcp automatically creates a structured finding page: control reference, severity level (High), SOC 2 requirement, repeat-finding status with expiration date, brief technical evidence summary, and MinIO URLs. The engineer reviews and provides the URL to the external auditor.
Total time for full process: ~8 mins. Manual effort would take about 3–4 hours spread over two days.

## Why this stack vs ChatGPT/Copilot

1. The SSH target is inaccessible via third-party infrastructure: ssh-exec uses the SSH protocol to connect to 10.2.0.3:22 from inside the Docker network (an RFC 1918 address with no public routes exposed). This means that even in an enterprise ChatGPT or Copilot environment, OpenAI and Microsoft have no mechanisms for connecting into our infrastructure. The tool that checks for unrestricted SSH access would require granting unrestricted SSH access to a third party.
2. The AWS credential model cannot move to a cloud product without introducing the finding it is trying to detect: The compose file mounts ~/.aws:/root/.aws:ro into MCPHub. The AWS SDK automatically obtains the STS tokens as they expire. However, granting equivalent access to a tool would involve exporting our long-term IAM keys, which is CIS 1.13/1.14 violation. The point here isn't about convenience, but it's a focus on security. It's impossible for the AWS credentials to leave the host in our scenario without compromising security.
3. Evidence has no auditable chain of custody through a third-party inference layer: SOC 2 Type II artifacts have to be verifiable: you have to know exactly what system they were collected from, what time and who did that, and what is the storage method for the artifact. In our case, the chain of custody goes end-to-end: Playwright creates a timestamped PDF; MinIO uploads it with ETag; the auditor pulls that PDF from there and verifies the ETag. With ChatGPT or Copilot, all artifacts go through their systems, and there is absolutely no way of independent verification anymore.
